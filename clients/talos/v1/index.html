真需求与伪需求

🧊 一、先定义“真需求”与“伪需求”

  • 真需求：用户有持续、明确、可支付的痛点，且现有方案无法很好解决；一旦产品可用，用户会自发购买并传播。
  • 伪需求：看似有场景，但痛点不够强、频率不够高、支付意愿不足、替代方案太多；即便产品做出来，也难以形成规模销售。

🏘️ 二、从应用场景看：真实痛点确实存在

✅ 典型痛点：

  1. 北美住宅区的雪清理负担极重——老年人、女性家庭、出差人员、物业管理方都深受其扰。
  2. 劳动力短缺严重——尤其加拿大、北部美国，人工除雪价格上涨且供应不足。
  3. 夜间/暴雪后清晨场景刚需——“一夜大雪后6点前必须清理车道”，这是真实的时间痛点。
  4. 电动+自动化趋势吻合政策方向——低噪音、零排放、可加入智能家居生态（Nest、Tesla、HomeKit等）。

→ 所以，从痛点角度看，这不是伪需求。

它是真实存在的生活问题。

💰 三、但从“商业化角度”看，大概率是局部真需求、广义伪需求

❌ 理由1：高昂的成本 vs. 低频使用

  • 大部分地区每年真正需要除雪的天数不到15天；
  • 用户很难为一个“每年用十几次的设备”支付 2000–5000 美元；
  • 低频 + 高单价 = 投资回报周期太长；
  • 人工除雪仍便宜（100–200美元/次），且灵活。

❌ 理由2：物理场景复杂度极高

  • 雪的密度、厚度、含冰量差异大；
  • 地形复杂（车道、草坪、边坡、门口台阶）；
  • GPS和激光雷达在大雪中易失效；
  • 积雪遮挡标志、边界模糊，导致路径规划困难；
  • 防冻、防滑、电池衰减、驱动扭矩、摄像头加热都需要复杂工程设计。

→ 工程难度 ≈ “波士顿动力+Roomba+扫雪机”的组合体。

这是一个技术上极难、商业上难以收回成本的领域。

❌ 理由3：消费习惯未形成

  • 人们对扫地机器人、割草机器人已经熟悉；
  • 但“扫雪机器人”认知极低，信任度差；
  • 除雪场景要求可靠性=命（清不干净就出不去），容错空间极小；
  • 因此它更像是**“科技玩具 + 小众奢侈品”**，而非大众家电。

🧠 七、战略判断总结

真需求不在“家用”，而在“无人作业服务化”。

未来3年，若你想让除雪机器人成为商业项目，不应卖机器，而应：

  1. 以“机器人除雪服务 / Robot-as-a-Service (RaaS)”模式卖工时；
  2. 做“多机协同 + 云调度 + 租赁”；
  3. 用政府项目 / 工业园示范 / 机场跑道清雪带动需求；
  4. 产品形态：履带底盘 + 模块化推雪 + 摄像头/雷达防撞 + 自动回充；
  5. 收费模式：按面积 / 次数 / 季节包年；
  6. 最后再下沉C端，用“家用除雪小车”做形象产品。

2. 场景设计对照表

S1 郊区独立屋车道（Suburban Driveway）

维度设计建议
底盘/驱动小型履带或 4×4 轮式，地间隙 80–120 mm，支持在 1 m 雪堆前抬头；转弯半径小，适合短车道。
执行机构标配推雪板（800–1000 mm），带左右 30° 偏转 + 升降；可选小型抛雪头用于把雪抛到院外一侧。
电源与续航48 V / 4–6 kWh，能在 -20℃ 有效工作 60–90 分钟；要求回站自动充。
感知/传感器前超声/ToF 检测车库门、车辆；IMU 防翻；夜间 LED 补光；短距边界检测防冲到草坪。
控制与任务策略“定时多趟”策略：夜里每 1–2 h 清一次，只走主车道；6:00 做一次全宽清扫。设高优先级走廊。
基站/部署要求放在屋檐下或车库门侧边，带融雪板和对接充电触点，防被雪埋；Wi-Fi 覆盖。

S2 市区人行道（Urban Sidewalk）

维度设计建议
底盘/驱动窄体底盘（≤600 mm），小履带或四轮独立；要求高机动、能原地掉头；防止压到草坪。
执行机构滚刷 + 小刮板，适合 2–10 cm 踩实小雪；刮板要有弹性，不破坏人行道砖。
电源与续航48 V / 3–4 kWh 就够，重点是低噪音和轻量。
感知/传感器侧向/前向短距传感，识别路牌杆、邮箱、垃圾桶；低光摄像头；可加简单语音提示。
控制与任务策略“边界跟随”+“双向往返”模式，一条 1.5 m 道一次做完；可设“上班前窗口”高优先级；遇人减速绕行。
基站/部署要求可放在建筑物入口侧墙下，要求占地小、防盗。

S3 小区内部道路 / 停车场通道

维度设计建议
底盘/驱动中号履带，牵引力更高，带防滑齿；能跨过停车场小坎；需要 1.2–1.4 m 作业宽度。
执行机构宽推雪板 + 可选抛雪头；板要有“浮动”以适应停车场起伏；推到指定堆雪区。
电源与续航48–72 V / 6–8 kWh，保证一次能扫完 2–3 条车道后回充。
感知/传感器前激光/雷达看停放的车；超声识别路沿；IMU 防侧倾。
控制与任务策略分区任务：车道优先、停车位次之；夜间多趟巡逻防再堆积；需要“已扫区域标记”。
基站/部署要求小区内固定点，可能要拉户外 AC；要求 IP65 外壳，防住户误碰。

S4 商业广场/超市停车场

维度设计建议
底盘/驱动中大型履带或轮-履混合，要求长时间低速高扭；底盘要抗雪泥 + 防石子。
执行机构抛雪机头（双螺旋+叶轮）为主，推雪板为辅；要能把雪送到远端集中堆放区。
电源与续航72 V / 8–12 kWh，最好支持换电；因为面积大，单次电量可能不够。
感知/传感器360° 雷达/多雷达抗遮挡，识别停着的车、购物车；夜间强光照明。
控制与任务策略需要任务调度：先扫车流主干线，后扫停车位；遇到车辆临停要局部重规划；支持后台远程看图。
基站/部署要求可能需要多基站布点；基站要有大功率加热融雪和防撞护栏。

S5 屋顶平面除雪

维度设计建议
底盘/驱动超轻量小履带，单位面积压强低；底部软垫，防伤屋面/光伏板；限高限重。
执行机构大直径软滚刷/轻刮板，推向落雪口；不能用太重的抛雪机。
电源与续航24–48 V / 2–3 kWh，任务短，但要能低温启动。
感知/传感器必配“坠落边缘检测”（下视 ToF/超声）；IMU 检测异常倾斜；前上摄像头可选。
控制与任务策略走“网格清扫”或“蛇形清扫”；清到屋边要自动留安全距离；可定时化解积雪。
基站/部署要求屋顶要有固定停靠点，遮雪，最好接市电；基站要锚固防风。

S6 工业园区通道（有沟槽/轨道）

维度设计建议
底盘/驱动高通过性履带，带防卡设计；底部钢板防雪硬块击打；电机冗余更高。
执行机构抛雪机头 + 破冰/铲刀前缘；需要较大扭矩来应对硬雪/冰块。
电源与续航72 V / 8–10 kWh，甚至支持外接 DC；工业区作业时间长。
感知/传感器前雷达+下视传感，识别沟槽/轨道；IMU 保持姿态；可选工业 RFID 做定位。
控制与任务策略任务按通道 ID 执行；检测到沟槽则抬高执行头；需要“脱困模式”。
基站/部署要求工业区可设固定“暖棚+充电”点，内置工业加热器，机器人作业完回棚除冰。

S7 乡村长车道/坡道

维度设计建议
底盘/驱动履带必选，配重后置，保证上坡牵引；IMU 实时测坡度做功率限制。
执行机构宽推雪板 + 抛雪口偏转，可把雪往下风口抛；必要时分两趟推（上行+下行）。
电源与续航72 V / 6–8 kWh，长距离消耗大；可设中途回站点。
感知/传感器IMU+里程计为主；前照灯+摄像头辅助；坡下障碍超声探测。
控制与任务策略“上下行”两种路径；上行时限流防打滑；暴雪时缩短出动周期。
基站/部署要求基站应设在坡顶/有棚处，防止被雪堵门；要有加热。

S8 暴风雪 + 吹雪再堆积（Blizzard & Drift）

维度设计建议
底盘/驱动中重型履带，防风雪包覆好，顶部做防雪檐；所有传感器做加热/防雾。
执行机构抛雪机头为主，推雪板辅助开路；要求能做“分层吃雪”——先高后低。
电源与续航72 V / 10–12 kWh，+ 必须有可频繁回站补电的策略。
感知/传感器雷达优先，光学只是辅助；风向传感器；前下 ToF 判断雪墙；温度/能见度监测。
控制与任务策略典型策略就是你刚说的那个：整夜多趟巡逻，优先保通车道；根据风向自动调整抛雪方向；被埋自动求助。
基站/部署要求必须是封闭式或半封闭式基站，有大功率加热和高位定位标记，防止基站被雪掩。

S9 冰雨后结冰场景

维度设计建议
底盘/驱动履带或带防滑钉的轮式，防止在冰面打滑；底盘要能承受振动。
执行机构推雪板前缘加加热/震动/硬刃；后面再用滚刷清理残雪；必要时用抛雪把碎冰送走。
电源与续航48–72 V / 5–6 kWh，额外给前缘加热预留功率。
感知/传感器地面滑动检测（通过电机电流+速度差）；温度传感；前超声防止撞到冰坎。
控制与任务策略先破冰/起拱，再低速推走；软件里要允许“同一条线反复走 2~3 次”。
基站/部署要求基站底板要加热，否则机器人回去也会上冰。

S10 半公共场地（学校、社区中心）

维度设计建议
底盘/驱动中型履带，兼顾灵活与通过；做防撞包角。
执行机构推雪板为主，部分分区用抛雪；靠近门廊用滚刷。
电源与续航48–72 V / 6–8 kWh；停课日可慢速长时间作业。
感知/传感器360° 障碍检测，室外摄像头，热释电/毫米波识别人；身份识别可选。
控制与任务策略中心式任务调度：管理员下发“今天清 1 号门→操场→停车区”；机器人按分区逐个执行；有人靠近降速或暂停。
基站/部署要求通常放在物业/机房外墙边，市电充电，需防学生误触；支持远程升级和日志上传。

3. 怎么用这张表

  1. 机械团队看“底盘/执行机构”两列，确定要做几种平台。
  2. 电气团队看“电源与续航 + 基站”，确定要不要加热、要不要72 V、要不要快换电。
  3. 算法/软件团队看“感知 + 控制策略”，确定最少要适配哪几类路径规划模式。
  4. 你要是做产品型机器人，可以从这 10 个选 3 个高频场景（S1、S3、S8）做成 V1.0 支持列表，其他写成“计划支持”。
