0. 全局设计原则与工程冻结（Global Principles & Freeze）

0.1 全球统一内核（Global Core）

  • EnergizeOS™ 采用单一全球核心代码库
  • 所有核心控制逻辑（并离网、PCC 控制、策略裁决、审计）仅存在一套实现
  • 禁止任何区域 fork 核心逻辑
Global Core
├─ Core Control Engine
├─ Strategy Engine
├─ Audit & Event System
├─ Security & Permission
└─ Local-First Runtime

0.2 区域能力插件化（Regional Add-ons）

  • 区域差异仅允许以 插件 / 模块 / 配置 形式加载
  • 区域模块 不得修改或侵入 Global Core
  • 区域模块具备 可选安装、可卸载、可版本化 特性
Global Core
└─ Regional Add-ons
   ├─ CN Plugin
   ├─ US Plugin
   ├─ EU Plugin
   └─ AU / SEA Plugin

0.3 核心控制逻辑不可分叉

  • 并–离–并状态机：全球唯一
  • PCC 合闸 / 跳闸控制模型：全球唯一
  • 策略执行与裁决顺序：全球唯一

禁止行为：

  • 针对单一项目定制核心逻辑
  • 为某区域复制一套“特殊版本内核”
  • 在区域插件中实现“影子控制逻辑”

0.4 工程优先级冻结

系统设计优先级固定如下（不可调整）：

稳定性
 > 可审计性
   > 可复现性
     > 功能丰富度
  • 任何功能若影响系统稳定性 → 直接否决
  • 任何控制动作若不可被审计 → 禁止上线
  • UI、美观、自动化便利性 永远低于工程确定性

0.5 Engineer of Record 责任冻结

  • EnergizeOS™ 不承担 Engineer of Record（EoR）
  • 不提供：
    • 最终施工图
    • 保护定值
    • Utility 合规背书
    • 并网审批文件

系统定位：

  • 控制理念
  • 接口定义
  • 判据框架
  • 工程级参考实现

0.6 版本演进冻结规则

  • 当前目录树为 宪法级冻结结构
  • 后续版本：
    • ✅ 允许：向下细化、实现、补充
    • ❌ 禁止：重构、推翻、新增顶层章节

1. 系统边界与责任划分（Control Boundary & Responsibility）

1.1 系统角色边界定义

角色核心职责
EMS（EnergizeOS™）判据裁决、策略执行、控制授权
AIPP并网保护、主网异常检测
EPC / Panel Builder设备选型、接线、施工
Utility / AHJ并网审批、法规裁决

1.2 合闸控制权唯一性（EMS Only）

  • 合闸（Close）仅允许 EMS 发起
  • 任意非 EMS 合闸路径 → 非法路径
合法路径：
EMS → DO2 → Close Coil → PCC Breaker

非法路径：
Relay Auto-Close
Local Push Button
PCS / Inverter 自行合闸

1.3 跳闸控制的冗余与优先级

跳闸允许多路径，但具备明确优先级：

Priority 1: Protection Relay (AIPP)
Priority 2: EMS 冗余跳闸
Priority 3: UVT / 控制电源失效
Priority 4: 人工紧急断开
  • 跳闸路径 允许冗余
  • 合闸路径 绝对禁止冗余

1.4 非 EMS 合闸路径禁止声明

明确禁止以下行为：

  • 继电器 Auto-Reclose
  • PCS 内部并网自恢复
  • 外部 SCADA 直接控制 Close Coil
  • 本地 HMI 绕过 EMS 授权

后果：

  • 系统进入 Unsupported State
  • EnergizeOS™ 拒绝调试 / 策略支持

1.5 合规责任与法律边界

  • 并网合规责任：EPC / PE / Utility
  • EnergizeOS™ 输出不构成：
    • 法规承诺
    • 合规担保
    • 安全背书

1.6 技术输出的免责与适用范围

EnergizeOS™ 所有输出均属于：

  • 控制逻辑参考
  • 系统架构示意
  • 判据与策略模板

最终配置与现场行为责任归属：

  • EPC
  • Panel Builder
  • Licensed PE

2. EMS 总体系统架构（System Architecture）

2.1 Global Core + Regional Add-ons 架构

  • 系统由 全球统一内核（Global Core） 与 区域插件（Regional Add-ons） 组成
  • Global Core 始终常驻、本地运行
  • Regional Add-ons 按区域、法规、商业需求加载
EMS Runtime
├─ Global Core (Always On)
│  ├─ Core Control Engine
│  ├─ Strategy Engine
│  ├─ Audit & Event Engine
│  ├─ Security & Permission
│  └─ Local Runtime Manager
│
└─ Regional Add-ons (Optional)
   ├─ CN / US / EU / AU / SEA
   └─ Compliance / Tariff / DR Plugins

2.2 控制链路结构（Control Flow）

  • EMS 控制链路为 单向可追溯闭环
  • 所有控制动作必须具备 感知来源、判据依据、执行记录、反馈确认
感知 (Sense)
 → 判据 (Decision)
   → 控制 (Actuation)
     → 状态反馈 (Feedback)
  • 任一环节缺失 → 控制动作无效

2.3 通信链路结构（Communication Flow）

  • 南向通信：设备级、实时控制
  • 北向通信：监控、调度、集成
Northbound
├─ SCADA / DMS
└─ Cloud (Optional)

Southbound
├─ PCS / BESS
├─ PV Inverter
├─ DG Controller
├─ Protection Relay
└─ Meter / Sensor

2.4 冗余与失效模式设计

  • EMS 采用 Fail-safe + Fail-degraded 组合设计
  • 任一模块失效时：
    • 不引发非法合闸
    • 不破坏系统安全状态
Failure Scenario
├─ Communication Loss → Freeze / Degrade
├─ EMS Crash → UVT / Relay Protection
└─ Partial Data Loss → Conservative Strategy

2.5 本地优先、离线可运行架构

  • EMS 不依赖云端运行
  • 云端仅用于：
    • 策略分发
    • 远程运维
    • 数据汇总
Offline Mode
├─ Core Control
├─ Strategy Execution
├─ Logging & Audit
└─ Local HMI

3. 核心控制引擎（Core Control Engine）

3.1 并–离–并状态机（Grid ↔ Island ↔ Grid）

  • 核心控制逻辑采用 显式状态机
  • 任一时刻系统仅处于一个合法状态
Grid-Connected
   ↓ (Grid Abnormal)
Islanded
   ↓ (Sync Ready)
Grid-Reconnected

3.2 状态定义与迁移条件

  • 状态迁移必须满足 完整判据集合
  • 不允许“强制跳状态”
Transition Conditions
├─ Voltage / Frequency
├─ Protection Relay Status
├─ PCC Breaker Feedback
└─ Internal Interlock (C1–C7)

3.3 主网异常检测

  • 主网异常由 保护装置与冗余感知路径 判定
  • EMS 不直接承担一次保护功能
Grid Fault Detection
├─ AIPP Relay (Primary)
└─ Meter / EMS Monitoring (Secondary)

3.4 孤岛切入流程

  • 孤岛切入为 保护优先、控制跟随
  • EMS 不抢占继电器跳闸控制权
Grid Fault
 → Relay Trip
   → PCC Open Confirmed
     → EMS Island Mode Enable

3.5 再并网同步与合闸控制

  • 再并网必须满足 同步 + 授权 + 互锁
  • 合闸仅由 EMS 执行
Reconnection Flow
├─ Sync Check (V / f / Phase)
├─ Interlock Check (C1–C7)
├─ EMS Close Command (DO2)
└─ 52a Feedback Confirmation

3.6 PCC 主断路器 I/O 模型

DO1 → Shunt Trip Coil
DO2 → Close Coil
DI1 → 52a (Closed)
DI2 → 52b (Open)
  • I/O 定义 全球冻结
  • 不允许项目级重定义

3.7 合闸互锁判据（C1–C7）

  • 合闸前必须 全部满足
  • 任一判据失败 → 合闸禁止
Interlock Gate
├─ Grid Healthy
├─ Sync Ready
├─ No Active Trip
├─ EMS Authorized
├─ Breaker Ready
├─ Power Source Ready
└─ No Lockout

3.8 黑启动与系统恢复序列

  • 黑启动为 显式受控流程
  • 禁止隐式自启动
Black Start
├─ Control Power Ready
├─ EMS Boot
├─ Local Source Enable
└─ Load Restoration Sequence

3.9 UVT / UPS / 控制电源失效链

  • 控制电源失效 → 系统必须跳闸
  • UVT 作为 最终物理 Fail-safe
UPS Loss
 → UVT De-energized
   → Breaker Trip

3.10 防误控、防越权、防非法合闸

  • 所有控制命令需满足：
    • 身份校验
    • 状态校验
    • 判据校验
    • 审计记录
Control Command
├─ Auth Check
├─ State Check
├─ Interlock Check
└─ Audit Log

4. 多电源协调与能量调度（Multi-Source Coordination）

4.1 多电源统一协调原则

  • EMS 作为唯一调度裁决层
  • 各电源控制器（PCS / PV / DG）仅执行本地一次控制
  • EMS 不替代设备内部保护与底层控制
EMS (Supervisory)
├─ Dispatch Decision
└─ Power Setpoints
   ├─ BESS
   ├─ PV
   └─ DG

4.2 BESS 管理（Battery Energy Storage System）

  • BESS 为核心可调度资源
  • EMS 负责能量层与策略层约束
BESS Control Scope
├─ SOC / SOH Monitoring
├─ Charge / Discharge Limits
├─ Reserve Margin
└─ Lifetime Constraints

关键约束（必须同时满足）：

  • SOC 上下限
  • 备用容量保留（Island / Black Start）
  • 循环次数与功率斜率限制

4.3 PV 协调控制

  • PV 以可再生优先为原则
  • EMS 仅在必要时进行限发或跟随控制
PV Modes
├─ Grid-Connected
│  ├─ MPPT / Export Limit
│  └─ Curtailment
└─ Islanded
   ├─ Grid-Forming Follower
   └─ Frequency / Power Following

4.4 DG 协调控制（Diesel Generator）

  • DG 作为备用 / 长时支撑电源
  • EMS 约束其非经济、非必要运行
DG Constraints
├─ Start / Stop Logic
├─ Minimum Run Time
├─ Minimum Loading
└─ Cooldown / Maintenance
  • 禁止频繁启停
  • 禁止低负载长时间运行

4.5 关键负荷优先级与切除机制

  • 负荷按等级与可中断性分组
  • EMS 仅在必要时执行切除
Load Priority
├─ Tier 1: Critical (Never Shed)
├─ Tier 2: Important
└─ Tier 3: Non-Critical
  • 切除动作必须记录并可审计

5. 策略系统设计与执行机制（Strategy System）

5.1 策略系统总体原则

  • 策略 ≠ 控制逻辑
  • 策略运行在 Core Control Engine 之上
  • 策略不允许绕过底层安全判据
Control Stack
├─ Protection & Interlock (Non-bypassable)
├─ Core Control Engine
└─ Strategy Layer (Configurable)

5.2 策略注册与生命周期管理

Strategy Lifecycle
├─ Register
├─ Configure
├─ Enable / Disable
├─ Execute
└─ Retire

注册信息（强制）：

  • 策略名称 / ID
  • 版本号
  • 适用条件
  • 互斥关系

5.3 策略执行引擎

  • 策略执行为 事件驱动
  • 不允许周期性盲执行
Trigger Sources
├─ State Change
├─ Time Window
├─ Threshold Event
└─ External Signal

5.4 策略优先级与裁决机制

  • 多策略同时满足时：
    • 按优先级排序
    • 按互斥关系裁决
Strategy Arbitration
├─ Priority Level
├─ Mutual Exclusion
└─ Safety Override
  • 安全相关策略优先级最高
